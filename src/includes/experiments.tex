\subsection{E1: Deterministic Merkle Commitment}\label{subsec:exp-deterministic-merkle}
\textbf{Hypothesis:} For the same set of canonical facts, the daily commitment is deterministic: identical Merkle roots and byte-identical \texttt{day.bin} (same \ac{SHA}-256).

\textbf{Method:} We compute Merkle roots using hash-sorted leaves (\ac{ADR}--003) over a golden corpus of facts and frames derived from \texttt{tests/fixtures/}.
We vary input size (empty set, single fact, odd count, power-of-2 count) and randomize file ordering to test order independence.
We repeat the computation across the Python implementation and the Rust implementation (\emph{trackone-core}), comparing results bit-for-bit under \ac{CI}.

\textbf{Result:} For all tested cases, Merkle roots match across implementations and across randomized file orderings.
For identical inputs, \texttt{day.bin} is byte-identical, yielding the same \ac{SHA}-256 digest.

\subsection{E2: Cryptographic Correctness and Tamper Detection}\label{subsec:exp-crypto-correctness}
\textbf{Hypothesis:} The gateway cryptographic pipeline is correct with respect to its known-answer vectors, and it rejects tampered frames with explicit failure modes.

\textbf{Method:} We verify deterministic \ac{AEAD} test vectors (XChaCha20-Poly1305) and run round-trip tests for X25519, \ac{HKDF} (RFC~5869), XChaCha20-Poly1305, and Ed25519.
We perform negative tests by mutating ciphertext, tags, \ac{AAD}, and nonces, and by presenting frames from unknown devices.
Property-based tests stress TLV parsing against malformed and truncated inputs.

\textbf{Result:} All vector and round-trip tests pass.
All tampering scenarios are rejected (tag mismatch, nonce validation failure, unknown device), and malformed inputs fail closed with explicit errors.

\subsection{E3: Replay Window Enforcement}\label{subsec:exp-replay-window}
\textbf{Hypothesis:} The framed ingest policy enforces an anti-replay sliding window, rejecting duplicates and out-of-window frames while tolerating reordering within the window.

\textbf{Method:} We execute attack scenarios against the frame counter (\texttt{fc}): monotonic acceptance, duplicate replay, out-of-window jumps, tolerated reordering, persistence across restart, and boundary conditions at the window edge.

\textbf{Result:} Duplicates and out-of-window frames are rejected; frames within the window are accepted even if they arrive out-of-order.
Persistence prevents replay after restart.

\subsection{E4: End-to-End Verification and Anchoring}\label{subsec:exp-end-to-end}
\textbf{Hypothesis:} The end-to-end pipeline produces a verifiable daily commitment that can be independently recomputed and checked, with optional external anchoring when artifacts are available.

\begin{figure}[H]
\centering
\includegraphics[width=0.97\linewidth]{figs/exp_end_to_end}
\caption{End-to-end experiment flow: from framed ingest through batching, anchoring, and independent verification.}
\label{fig:exp-end-to-end}
\end{figure}

\textbf{Method:} We run the end-to-end test flow:
\begin{enumerate}[nosep]
  \item \texttt{pod\_sim --framed} generates encrypted frames.
  \item \texttt{frame\_verifier} decrypts and validates authenticity.
  \item \texttt{merkle\_batcher} builds the Merkle tree over canonical commitment bytes.
  \item \texttt{ots\_anchor} produces an \ac{OTS} proof (real or placeholder, depending on mode).
  \item \texttt{verify\_cli} recomputes the root and verifies equality and proof structure.
\end{enumerate}
We additionally validate (a) peer attestation over day roots (Ed25519 signatures over a canonical message) and (b) optional \ac{RFC}~3161 \ac{TSA} responses when present.

\textbf{Result:} The pipeline exits successfully and the verifier recomputes the same root.
Peer attestation verifies for the configured quorum, and invalid contexts/roots are rejected.
When \texttt{*.tsr} artifacts are present, \ac{RFC}~3161 tokens parse and match the day root message imprint.

\subsection{E5: Embedded Firmware Viability}\label{subsec:exp-embedded-viability}
\textbf{Hypothesis:} The pod frame builder is feasible for L0-class \ac{MCU}s in terms of code size and stack/RAM usage.

\textbf{Method:} We run stress harnesses on host and QEMU targets (Cortex-M4 and Cortex-M0+), repeatedly building frames and checking stack guards.

\textbf{Result:} The stress harness runs complete without stack guard violations.
Measured code size and RAM headroom remain within the STM32L0 budget envelope for framing and transport (cryptographic throughput is not claimed by the stress harness).

\subsection{E6: Pipeline Performance and Storage}\label{subsec:exp-performance}
\textbf{Hypothesis:} The pipeline completes well within interactive operator workflows for typical day sizes, and storage overhead scales linearly with measurement volume.

\textbf{Method:} We measure wall time on Ubuntu~24.04 (Python~3.13, pytest runtime) and account for the size of day artifacts for 10 devices with 100 frames each.

\textbf{Result:}
\begin{center}
\begin{tabular}{lrr}
  \hline
  Operation & 10 frames & 100 frames \\
  \hline
  \texttt{pod\_sim --framed} & $<0.1$ s & $<0.2$ s \\
  \texttt{frame\_verifier} (\ac{AEAD}) & $<0.1$ s & $<0.5$ s \\
  \texttt{merkle\_batcher} & $<0.1$ s & $<0.5$ s \\
  \texttt{verify\_cli} & $<0.1$ s & $<0.5$ s \\
  \textbf{Full test suite (200+ tests)} & \multicolumn{2}{c}{\textbf{$\sim$20\,s}} \\
  \hline
\end{tabular}
\end{center}

Storage overhead per day (10 devices, 100 frames each):
\begin{itemize}[nosep]
  \item Frames (ND\ac{JSON}): $\sim$200 KB
  \item Facts (canonical): $\sim$150 KB
  \item Block headers: $\sim$10 KB
  \item Day blob: $\sim$15 KB
  \item \ac{OTS} proof: $\sim$5 KB (pending), $\sim$10 KB (confirmed)
\end{itemize}

Total daily storage is approximately 390~KB per 1{,}000 measurements and scales linearly with device count and sampling rate.

\subsection{E7: Stationary Calendar Evaluation}\label{subsec:exp-stationary-calendar}
\textbf{Hypothesis:} A stationary calendar mode provides deterministic local proof generation under outage conditions while preserving an explicit "pending public anchor" lifecycle.

\begin{figure}[H]
\centering
\includegraphics[width=0.97\linewidth]{figs/exp_stationary_calendar}
\caption{Stationary calendar evaluation: deterministic local proofs under outages, with an explicit upgrade path to public Bitcoin attestation.}
\label{fig:exp-stationary-calendar}
\end{figure}

\textbf{Method:} We run \texttt{tox -e ots-cal} with a local calendar/stub deployment and execute a scheduled weekly ratchet profile that enables strict real-\ac{OTS} checks.

\textbf{Result:} Local proofs remain deterministic under \ac{CI}.
When public anchoring is unavailable, artifacts remain verifiable with an explicit pending-anchor status rather than silent success.

\paragraph{Experimental setup notes}
The experiments above are exercised via an automated test suite and \ac{CI} workflows.
Some tests (real \ac{OTS} proofs, \ac{RFC}~3161 \ac{TSA} verification) are conditional on external artifacts and are treated as opt-in checks to avoid non-deterministic failures while preserving realism.
