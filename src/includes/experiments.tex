\section{Experiments}

\subsection{Determinism (M\#0--M\#5)}
We verify deterministic Merkle root computation:
\begin{itemize}[nosep]
    \item Same input facts \textrightarrow{} identical \texttt{merkle\_root} (hex)
    \item Same input facts \textrightarrow{} byte-identical \texttt{day.bin} (SHA-256)
    \item Hash-sorted leaves ensure order independence: shuffled fact files \textrightarrow{} same root
\end{itemize}

Test coverage includes: empty set, single fact, odd count, power-of-2 count, random filename ordering, Unicode edge cases.

\subsection{Cryptographic Correctness (M\#4--M\#5)}
\textbf{Deterministic AEAD vectors:} XChaCha20-Poly1305 test vectors in \texttt{crypto\_test\_vectors.json} verified against PyNaCl bindings.

\textbf{Round-trip tests:}
\begin{itemize}[nosep]
    \item X25519: Key generation and shared secret agreement
    \item HKDF: Deterministic derivation per RFC 5869 and length/empty-edge cases
    \item XChaCha20-Poly1305: Encrypt/decrypt round-trip, authentication failure detection
    \item Ed25519: Sign/verify signatures, tamper and wrong-key rejection
\end{itemize}

\textbf{Property-based testing (Hypothesis):} TLV encoding robustness against truncated/malformed data; unknown tags ignored.

\subsection{Security Testing (M\#4--M\#5)}
Tamper detection tests validate AEAD integrity:
\begin{itemize}[nosep]
    \item \textbf{Ciphertext tampering}: Single-bit flip \textrightarrow{} tag verification fails
    \item \textbf{Tag tampering}: Modified authentication tag \textrightarrow{} rejection
    \item \textbf{AAD tampering}: Modified header fields \textrightarrow{} tag mismatch
    \item \textbf{Nonce validation}: Invalid nonce length (not 24 bytes) \textrightarrow{} rejected
    \item \textbf{Unknown device}: Frame from unprovisioned device \textrightarrow{} rejected
    \item \textbf{Frame structure}: Missing fields, malformed JSON, and empty files handled gracefully
\end{itemize}

All security tests pass with proper error messages and logging.

\subsection{Framed Ingest Robustness (M\#4)}
Replay window enforcement with attack scenarios:
\begin{itemize}[nosep]
    \item \textbf{Monotonic acceptance}: Increasing \texttt{fc} within window \textrightarrow{} accepted
    \item \textbf{Duplicate rejection}: Replay same frame \textrightarrow{} rejected with logged warning
    \item \textbf{Out-of-window rejection}: Jumps beyond window (64) \textrightarrow{} rejected
    \item \textbf{Reordering tolerance}: Frames within window but out of order \textrightarrow{} accepted
    \item \textbf{Restart persistence}: Device table persisted \textrightarrow{} duplicates rejected after restart
    \item \textbf{Boundary acceptance}: Exactly highest+window accepted; window+1 rejected
\end{itemize}

\subsection{End-to-End Pipeline (M\#4)}
Complete workflow validated by \texttt{test\_end\_to\_end\_pipeline}:
\begin{enumerate}[nosep]
    \item \texttt{pod\_sim --framed} generates encrypted frames
    \item \texttt{frame\_verifier} decrypts with XChaCha20-Poly1305, validates tags
    \item \texttt{merkle\_batcher} builds Merkle tree, validates schemas
    \item \texttt{ots\_anchor} creates OTS proof (placeholder or real)
    \item \texttt{verify\_cli} recomputes root, verifies match and OTS timestamp
\end{enumerate}

Exit code 0 confirms entire chain is cryptographically verifiable.

\subsection{Peer Attestation (M\#5)}
We validate \emph{peer attestation} of daily Merkle roots using Ed25519:
\begin{itemize}[nosep]
    \item Deterministic message format: \texttt{trackone:day-root:v1|<site>|<day>|<root>}
    \item Signature generation via \texttt{scripts/gateway/peer\_attestation.py}
    \item Quorum checks: $k$ of $n$ signatures required (configurable fixture)
    \item Negative tests: wrong context, wrong root, unknown public key \textrightarrow{} verification fails
\end{itemize}

\subsection{TSA Verification (M\#5)}
Optional parallel RFC~3161 TSA responses (\texttt{*.tsr}) are validated when present:
\begin{itemize}[nosep]
    \item Timestamp token parsed; message imprint compared to daily Merkle root
    \item Certificate chain and signing time checked (skipped when offline)
    \item Tests are optional and auto-skip without \texttt{tsr} artifacts
\end{itemize}

\subsection{Performance (M\#4)}
Wall-time measurements on Ubuntu 24.04, Python 3.13, pytest runtime:

\begin{tabular}{lrr}
    \hline
    Operation & 10 frames & 100 frames \\
    \hline
    \texttt{pod\_sim --framed} & $<0.1$ s & $<0.2$ s \\
    \texttt{frame\_verifier} (AEAD) & $<0.1$ s & $<0.5$ s \\
    \texttt{merkle\_batcher} & $<0.1$ s & $<0.5$ s \\
    \texttt{verify\_cli} & $<0.1$ s & $<0.5$ s \\
    \textbf{Full test suite (200+ tests)} & \multicolumn{2}{c}{\textbf{$\sim$20 s}} \\
    \hline
\end{tabular}

Storage overhead per day (10 devices, 100 frames each):
\begin{itemize}[nosep]
    \item Frames (NDJSON): $\sim$200 KB
    \item Facts (canonical): $\sim$150 KB
    \item Block headers: $\sim$10 KB
    \item Day blob: $\sim$15 KB
    \item OTS proof: $\sim$5 KB (pending), $\sim$10 KB (confirmed)
\end{itemize}

\textbf{Total daily storage: $\sim$390 KB for 1{,}000 measurements.} Scales linearly with device count and sampling rate.

\subsection{Benchmarking plan (ADR-011)}
We will introduce repeatable micro-benchmarks using pytest markers and a small harness:
\begin{itemize}[nosep]
    \item AEAD throughput (encrypt/decrypt) over payload sizes (16B\,$\ldots$\,4KB)
    \item Merkle batching throughput vs. number of facts (n = 10\,$\ldots$\,100k)
    \item Canonical JSON serialization cost per fact
    \item End-to-end pipeline latency for typical day sizes (site\_demo)
\end{itemize}
Benchmarks will emit CSV under \texttt{out/bench/} and produce plots for the report.

\subsection{Skip conditions and external artifacts}
Some tests are intentionally skipped when optional artifacts are unavailable:
\begin{itemize}[nosep]
    \item Deterministic crypto vectors: \texttt{toolset/unified/crypto\_test\_vectors.json} not present \textrightarrow{} vector-matching tests are skipped
    \item Real OTS proofs: missing \texttt{ots} binary or Bitcoin headers \textrightarrow{} real integration tests are skipped
    \item Local artifacts (e.g., \texttt{out/site\_demo/...}): present only on certain machines; tests auto-detect and enable if found
    \item Peer attestation fixtures (e.g., \texttt{*.peers.json}): generated on-the-fly or loaded when present
\end{itemize}
This avoids false negatives while keeping real-world checks opt-in and reproducible.
