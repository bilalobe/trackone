\section{Experiments}

\subsection{Determinism (M\#0)}
We verify deterministic Merkle root computation by running \texttt{merkle\_batcher.py} twice on the same fact set:
\begin{itemize}
  \item Same input facts → identical \texttt{merkle\_root} (hex)
  \item Same input facts → byte-identical \texttt{day.bin} (SHA-256)
  \item Hash-sorted leaves ensure order independence: shuffled fact files → same root
\end{itemize}

Test coverage includes: empty set, single fact, odd count, power-of-2 count, and random filename ordering.

\subsection{Framed Ingest Robustness (M\#1)}
We validate the replay window enforcement with simulated attack scenarios:
\begin{itemize}
  \item \textbf{Monotonic acceptance}: 10 frames with increasing \texttt{fc} (0..9) → all accepted
  \item \textbf{Duplicate rejection}: Replay same frame → rejected with "duplicate" warning
  \item \textbf{Out-of-window rejection}: Jump \texttt{fc} by +100 beyond window (64) → rejected
  \item \textbf{Reordering tolerance}: Frames arriving within window but out of order → accepted
\end{itemize}

The \texttt{frame\_verifier.py} maintains per-device state (\texttt{highest\_fc\_seen}, \texttt{seen\_set}) to enforce these policies.

\subsection{End-to-End Pipeline (M\#1)}
The complete M\#1 workflow is validated by \texttt{test\_end\_to\_end\_pipeline}:
\begin{enumerate}
  \item \texttt{pod\_sim --framed} generates 7 frames from pod-003
  \item \texttt{frame\_verifier} parses frames, accepts all (monotonic fc)
  \item \texttt{merkle\_batcher} builds Merkle tree, validates schemas
  \item \texttt{ots\_anchor} creates OTS proof (placeholder)
  \item \texttt{verify\_cli} recomputes root, verifies match and OTS
\end{enumerate}

Exit code 0 confirms the entire chain is verifiable.

\subsection{Performance (M\#1)}
Preliminary wall-time measurements on a development machine (Ubuntu 24.04, Python 3.11):

\begin{tabular}{lrr}
\hline
Operation & 10 frames & 100 frames \\
\hline
\texttt{pod\_sim --framed} & <0.1s & <0.2s \\
\texttt{frame\_verifier} & <0.1s & <0.5s \\
\texttt{merkle\_batcher} & <0.1s & <0.5s \\
\texttt{verify\_cli} & <0.1s & <0.5s \\
\textbf{Total pipeline} & \textbf{<0.5s} & \textbf{<2s} \\
\hline
\end{tabular}

File size overhead per day (10 devices, 100 frames each):
\begin{itemize}
  \item Frames (NDJSON): ~200 KB
  \item Facts (canonical): ~150 KB
  \item Block header: ~10 KB
  \item Day blob: ~15 KB
  \item OTS proof: ~5 KB (pending), ~10 KB (confirmed)
\end{itemize}

Total daily storage: ~390 KB for 1,000 measurements.
Scales linearly with device count and sampling rate.

\subsection{Schema Validation (M\#0 \& M\#1)}
All artifacts are validated against JSON schemas when \texttt{--validate-schemas} is passed:
\begin{itemize}
  \item \texttt{fact.schema.json}: Device telemetry format
  \item \texttt{block\_header.schema.json}: Merkle block header
  \item \texttt{day\_record.schema.json}: Daily record with \texttt{prev\_day\_root}
\end{itemize}

Warnings are printed for validation failures, but batching continues (non-blocking for M\#1).
