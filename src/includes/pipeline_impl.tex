\section{Gateway Pipeline Implementation}

\subsection{Framed ingest (M\#1--M\#5)}
The gateway receives framed telemetry from simulated devices (pods), performs cryptographic verification, replay detection, and emits canonical facts for Merkle batching and OpenTimestamps (OTS) anchoring.

\paragraph{Transport and frame shape}
Frames are emitted as NDJSON (one JSON object per line).
Each frame contains:
\begin{verbatim}
{
  "hdr":   { "dev_id": u16, "msg_type": u8, "fc": u32, "flags": u8 },
  "nonce": base64(24 bytes),
  "ct":    base64(XChaCha20-Poly1305 ciphertext),
  "tag":   base64(16 bytes authentication tag)
}
\end{verbatim}

\textbf{M\#4 implementation:} Full XChaCha20-Poly1305 AEAD with tag verification.
The gateway decrypts \texttt{ct} using the device's \texttt{ck\_up}, verifies the authentication tag, and validates AAD binding to header fields.

\paragraph{Replay window policy}
Per-device state in \texttt{device\_table.json} tracks:
\begin{itemize}[nosep]
    \item \texttt{highest\_fc\_seen}: Maximum accepted frame counter
    \item \texttt{last\_seen}: ISO 8601 timestamp of last valid frame
    \item \texttt{msg\_type}, \texttt{flags}: Last seen header fields
\end{itemize}

Acceptance criteria:
\begin{itemize}[nosep]
    \item First frame: $\texttt{fc} \ge 0$ accepted
    \item Subsequent: $0 < \texttt{fc} - \texttt{highest\_fc\_seen} \le 64$
    \item Duplicates and out-of-window frames rejected
\end{itemize}

Device table persistence ensures replay protection survives gateway restarts.

\paragraph{Device table schema v1.0 (ADR-006)}
Forward-only schema policy: no migrations, clean slate at milestone boundaries.
\begin{verbatim}
{
  "_meta": {
    "version": "1.0",
    "salt8": "<base64-8B>",
    "ck_up": "<base64-32B>",
    "site_id": "an-001"
  },
  "101": {
    "highest_fc_seen": 42,
    "last_seen": "2025-10-13T12:34:56Z",
    ...
  }
}\label{verb:verbatim}
\end{verbatim}

Schema validated at runtime with clear regeneration guidance on version mismatch.

\paragraph{Canonical facts}
Accepted frames are decrypted and written as canonical fact JSON:
\begin{verbatim}
{
  "device_id": "pod-101",
  "timestamp": "<gateway-utc-iso>",
  "nonce": "<base64-24B>",
  "payload": { ... decrypted payload JSON ... }
}
\end{verbatim}

Facts validated against \texttt{toolset/unified/schemas/fact.schema.json}.
Canonicalization: sorted keys, UTF-8, no whitespace (ADR-003).

\paragraph{Batching and daily artifacts}
Deterministic Merkle tree construction:
\begin{enumerate}[nosep]
    \item Compute SHA-256 of each canonical fact
    \item Sort leaf hashes lexicographically (order independence)
    \item Build binary Merkle tree (duplicate last leaf if odd count)
    \item Emit block header with \texttt{merkle\_root}, \texttt{count}, \texttt{leaf\_hashes}
    \item Build day record with \texttt{prev\_day\_root} for chaining (zero bytes for genesis day)
    \item Write canonical day blob to \texttt{day/YYYY-MM-DD.bin}
\end{enumerate}

Both block headers and day records validated against JSON schemas when \texttt{--validate-schemas} flag is set.

\paragraph{Anchoring and verification}
Day blob anchored with OpenTimestamps (OTS). Optionally, parallel RFC~3161 TSA anchoring stores \texttt{.tsr} responses adjacent to \texttt{.ots} proofs (ADR-015).
Verification recomputes Merkle root from facts directory, compares to recorded block header, and verifies OTS and/or TSA proofs. In addition, \emph{peer attestation} provides immediate provenance: operators sign the daily root with Ed25519 and publish a quorum in \texttt{day/<YYYY-MM-DD>.peers.json}.

\paragraph{Quick start (M\#5)}
\begin{verbatim}
# Run complete pipeline
make run

# Run individual components
make test           # 200+ tests, ~85% measured coverage (local runs may skip optional OTS/TSA tests)
make test-cov       # with coverage report
make check          # lint + format + test
make ci             # full CI simulation
\end{verbatim}

The M\#5 pipeline: \texttt{pod\_sim --framed} $\rightarrow$ \texttt{frame\_verifier} (real AEAD) $\rightarrow$ \texttt{merkle\_batcher --validate-schemas} $\rightarrow$ \texttt{ots\_anchor} (and optionally TSA) $\rightarrow$ \texttt{peer\_attestation} $\rightarrow$ \texttt{verify\_cli}

Pipeline fails fast if any verification step does not succeed.

\subsection{Projection from TrackOne facts to SensorThings}
The ingestion pipeline operates entirely on the internal TrackOne schema (\texttt{Fact}, \texttt{EnvFact}, day blobs, Merkle roots, OTS metadata). Only after daily artifacts are verified do we populate an external OGC SensorThings store, ensuring the public API remains a projection with cryptographic guarantees.

\begin{enumerate}[nosep]
  \item Canonical facts are written to disk and committed to the ledger (day Merkle root + OTS proof).
  \item A gateway service scans verified \texttt{EnvFact} records and populates SensorThings entities:
        \begin{itemize}[nosep]
          \item \emph{Things} and \emph{Locations} per pod/site derived from \texttt{device\_id} metadata.
          \item \emph{Datastreams} per \texttt{SampleType} (ambient air temperature, relative humidity, supply voltage, etc.).
          \item \emph{Observations} per day (or sub-day window) using \texttt{phenomenon\_time\_*} and the summarised values.
        \end{itemize}
  \item Downstream tools (GIS clients, dashboards) consume SensorThings only and never see raw frames or unverifiable intermediate states.
\end{enumerate}

By describing SensorThings as a projection rather than the system of record, we can rebuild the public API from immutable TrackOne artifacts whenever query needs evolve without touching the pod firmware or verification logic.
