\section{Conclusion and Roadmap}

\subsection{M\#4 Achievements}

Track1 (Barnacle Sentinel) has completed Milestone \#4, delivering a production-oriented, verifiable telemetry pipeline with Bitcoin-anchored timestamp proofs.
The system demonstrates:

\textbf{End-to-end cryptographic security:}
\begin{itemize}[nosep]
    \item XChaCha20-Poly1305 AEAD encryption with 16-byte authentication tags
    \item X25519 ECDH key agreement with HKDF-SHA256 key derivation
    \item Ed25519 signature support for future device attestation
    \item Replay protection with configurable 64-frame sliding window
    \item Persistent state management across gateway restarts
\end{itemize}

\textbf{Verifiable batching and anchoring:}
\begin{itemize}[nosep]
    \item Deterministic canonical JSON outputs suitable for byte-for-byte reproducibility
    \item Hash-sorted Merkle tree construction (reduces ingestion-order bias)
    \item Daily OpenTimestamps anchoring to the Bitcoin blockchain (calendar attestations)
    \item Independent verification CLI for third-party auditors
\end{itemize}

\textbf{Production readiness (notes):}
\begin{itemize}[nosep]
    \item Test suite: 181 tests (2 skipped in some runs); coverage measurements currently report 83.8\% overall for gateway and pod simulation modules
    \item Test categories: unit, integration, security, edge cases, and property-based tests
    \item CI matrix: Python 3.11/3.12/3.13 (uv-based dependency management in CI as an experimental option)
    \item Static analysis: mypy (type checks), ruff (linting), bandit (security scanning)
    \item Git LFS used for large artifacts such as OTS proofs
\end{itemize}

The M\#4 milestone validates the core thesis: resource-constrained heritage sensors can achieve cryptographic verifiability without trusted intermediaries.
Bitcoin timestamping provides public auditability while payload confidentiality is preserved through AEAD encryption.

\subsection{Operational Considerations (clarified)}

\textbf{Deployment scenarios:}
\begin{itemize}[nosep]
    \item \textbf{Cloud-connected:} Full automation with Bitcoin P2P and public OTS calendars
    \item \textbf{Intermittent connectivity:} Queue OTS operations and batch during uplink windows
    \item \textbf{Air-gapped:} Manual OTS export/import via removable media; pre-signed header bundles for offline verification
\end{itemize}

\textbf{Scalability characteristics:}
\begin{itemize}[nosep]
    \item \textbf{Pod count:} Linear growth in frame ingestion (tested at small scale; further benchmarking planned)
    \item \textbf{Frame rate:} Configurable per-pod cadence (typical: one frame / 5 minutes)
    \item \textbf{Daily batch size:} $O(n \log n)$ Merkle computation (practical for $n < 10^6$ facts)
    \item \textbf{OTS overhead:} One proof per day regardless of fact count (fixed recurring cost)
\end{itemize}

\subsection{Research Contributions (hedged and clarified)}

This work presents Track1, a \textbf{practical, open reference implementation} that integrates well-known cryptographic components and timestamping workflows for a focused deployment niche: heritage sites requiring long-term data verifiability without vendor lock-in.

\textbf{Key insights:}
\begin{enumerate}[nosep]
    \item \textbf{Ultra-low power adaptation:} XChaCha20-Poly1305 AEAD with LoRa duty cycle constraints enables multi-year battery operation in typical configurations

    \item \textbf{Deterministic batching:} Canonical JSON and hash-sorted Merkle trees ensure reproducible roots for independent verification

    \item \textbf{Negligible marginal-cost anchoring:} Daily OpenTimestamps batching amortizes per-fact blockchain costs; after initial node/setup overhead the per-fact incremental cost is effectively negligible compared with commercial timestamping services\footnote{Initial Bitcoin node setup and header synchronization are one-time costs; ongoing node maintenance is small and amortized over many facts.}

    \item \textbf{Open implementation and reproducibility:} A documented reference implementation, ADRs, and test artifacts enable third-party auditors to reproduce verification results
\end{enumerate}

\textbf{Comparison with prior art (concise):}
\begin{itemize}[nosep]
    \item \textbf{Certificate Transparency / Trillian:} Merkle-based transparency model; Track1 adapts Merkle anchoring to IoT telemetry and uses Bitcoin as the public anchor rather than a dedicated CT log operator
    \item \textbf{Guardtime KSI:} KSI provides enterprise timestamping with different trust and licensing assumptions; Track1 is an open reference tied to Bitcoin for public attestations
    \item \textbf{Helium / blockchain-native LoRa networks:} Alternative cost and trust models; Track1 targets a different operational trade-off (daily batching, minimal per-device power)
    \item \textbf{Commercial IoT platforms:} Vendor-managed telemetry and logs; Track1 emphasizes independent verification and avoidance of vendor lock-in
\end{itemize}

Track1 targets a \textbf{well-defined niche}: publicly-auditable telemetry for heritage sites where curators need long-lived verifiability that survives organizational changes.

\subsection{Final Remarks (updated)}

Track1 achieves its practical goal: verifiable telemetry for resource-constrained sensors without trusted intermediaries.\ Milestone \#4 delivers a solid foundation with documented artifacts and reproducible verification steps.

Next steps before broad pilot deployments: increase unit-test coverage to meet the 85\% target, add a small ledger UI for auditors (M\#5), and harden operational runbooks for air-gapped and intermittent networks.

With 181 tests (2 skipped in some runs) and ongoing verification against live OpenTimestamps calendars, Track1 is positioned for pilot deployments in heritage sensor networks where long-term data integrity is paramount.
