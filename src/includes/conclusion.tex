\section{Conclusion and Roadmap}
\label{sec:conclusion}

\subsection{Summary of achievements}
\label{sec:achievements}

TrackOne delivers a production-informed, verifiable telemetry pipeline combining AEAD-protected uplinks, deterministic Merkle batching, and Bitcoin-backed timestamp proofs with optional peer attestation and RFC~3161 TSA support (ADR-015). The implementation and tests exercise the core requirements expressed in Section~\ref{sec:system-requirements} and the security/safety net described in Section~\ref{sec:security}.

\textbf{Evidence and integrity pipeline:}
\begin{itemize}[nosep]
    \item XChaCha20-Poly1305 AEAD with X25519/HKDF-SHA256 and a 64-frame replay window provide confidentiality and authenticity for framed uplink telemetry, with persistent state across gateway restarts.
    \item Canonical JSON and hash-sorted Merkle trees (ADR-003) make day roots reproducible by independent verifiers regardless of ingestion order.
    \item Daily OpenTimestamps anchoring to Bitcoin, plus a verification CLI, let third parties recompute roots from \texttt{facts/} and validate proofs without trusting the gateway operator.
    \item Peer attestation of day roots and optional TSA anchoring provide short-term provenance and multi-protocol resilience.
\end{itemize}

These outcomes validate the core thesis: resource-constrained subterranean pods can produce telemetry whose \emph{integrity and timing} are publicly auditable, while operational safety and actuation remain scoped to later tracks.

\subsection{Roadmap}
\label{sec:roadmap}

We organise follow-up work into near-term hardening (within Track One scope) and longer-term tracks for control and performance.

\textbf{Near-term hardening:}
\begin{itemize}[nosep]
    \item \textbf{Ledger and operator UX:} lightweight ledger views (CLI or web) for auditors to browse blocks, day records, proofs, and peer attestations.
    \item \textbf{Automated OTS workflow:} encode anchor and upgrade steps into a small, testable job (cron/systemd) with clear states (pending, anchored, upgraded) and notification hooks.
    \item \textbf{Outage and gap logging:} outage logger tracking expected pod cadence and surfacing missing-day events aligned with ADR-022 operational constraints.
    \item \textbf{Multi-day verification:} tooling to verify \texttt{prev\_day\_root} chains and report breaks or ambiguous anchors.
\end{itemize}

\textbf{Longer-term enhancements (Track Two / Track Three):}
\begin{itemize}[nosep]
    \item Key rotation and attestation (ADR-001, ADR-006) with Ed25519-based device attestation and revocation workflows.
    \item Air-gapped and intermittent deployment procedures: signed header bundles and removable-media workflows for offline verification.
    \item Performance-driven Rust core (ADR-017): gate Rust/PyO3 hot paths behind benchmarks (ADR-011) without changing Python APIs.
    \item Parallel anchoring and PKI productionisation (ADR-015) for deployments requiring PKI-backed timestamps in addition to Bitcoin proofs.
\end{itemize}

\subsection{Operational considerations}
\label{sec:operational-considerations}

Operationally, Track1 is ready for disciplined pilot deployments under the constraints in Section~\ref{sec:security} and the requirements captured in Section~\ref{sec:system-requirements}:
\begin{itemize}[nosep]
    \item \textbf{Deployment profiles:} cloud-connected gateways may run full Bitcoin + public calendars; intermittently connected gateways batch OTS operations; air-gapped sites rely on manual export/import of signed header bundles.
    \item \textbf{Sizing envelope:} the implementation supports small-to-medium pod fleets at typical cadences (e.g., one frame per 5 minutes per pod); Merkle computation remains practical for daily fact counts well below $10^6$.
    \item \textbf{Safety net:} CI, schemas, and the safety-net mechanisms in Section~\ref{sec:security} must remain enabled; Track1 artifacts should not be used as the sole basis for autonomous actuation without a separate safety case.
\end{itemize}

\subsection{Research contributions}
\label{sec:research-contributions}

This work presents Track1 as a \textbf{practical, open reference implementation} that integrates well-known cryptographic components and timestamping workflows for subterranean heritage and irrigation sensing where long-term verifiability and vendor independence are essential.

\textbf{Key insights:}
\begin{enumerate}[nosep]
    \item \textbf{Ultra-low power adaptation:} XChaCha20-Poly1305 AEAD with LoRa-friendly framing and duty-cycle constraints enables multi-year battery operation in typical scalar-telemetry configurations.
    \item \textbf{Deterministic batching:} Canonical JSON and hash-sorted Merkle trees ensure reproducible roots for independent verification, even under non-deterministic ingestion order and partial failures.
    \item \textbf{Negligible marginal-cost anchoring:} Daily OpenTimestamps batching amortizes per-fact blockchain costs; initial node/header sync is a one-time cost with small ongoing maintenance.
    \item \textbf{Open implementation and reproducibility:} documented reference implementation, ADRs, test artifacts, and this report enable third-party auditors to reproduce verification results and reason about trade-offs.
\end{enumerate}

\subsection{Final remarks and versioning}
\label{sec:final-remarks}

This release delivers a solid foundation with reproducible artifacts, ADRs, and a security/safety net to support pilot deployments under ADR-022. Short-term focus is on hardening (test coverage, ledger UX, runbooks); medium-term work prioritises measured, auditable releases and performance-driven decisions for when to introduce Rust components.

The strategy remains deliberate: maintain the evidence pipeline as a trustworthy base (Option~A), keep experimental or optional features documented in ADRs, and expand scope only when hardware, benchmarks, and safety analysis justify it.
