\section{Conclusion and Roadmap}

\subsection{M\#4 Achievements}

Track1 (Barnacle Sentinel) has successfully completed Milestone \#4, establishing a production-ready, verifiable telemetry pipeline with Bitcoin-anchored timestamp proofs. The system demonstrates:

\textbf{End-to-end cryptographic security:}
\begin{itemize}[nosep]
    \item XChaCha20-Poly1305 AEAD encryption with 16-byte authentication tags
    \item X25519 ECDH key agreement with HKDF-SHA256 key derivation
    \item Ed25519 signature support for future device attestation
    \item Replay protection with configurable 64-frame sliding window
    \item Persistent state management across gateway restarts
\end{itemize}

\textbf{Verifiable batching and anchoring:}
\begin{itemize}[nosep]
    \item Deterministic canonical JSON with 100\% reproducibility
    \item Hash-sorted Merkle tree construction (eliminates ingestion-order bias)
    \item Daily OpenTimestamps anchoring to Bitcoin blockchain
    \item Production proof verified against Bitcoin block 919384
    \item Independent verification CLI for third-party auditors
\end{itemize}

\textbf{Production readiness:}
\begin{itemize}[nosep]
    \item 158 tests with 85\% code coverage across gateway and pod simulation modules
    \item Comprehensive test categories: unit, integration, security, edge cases, property-based
    \item CI/CD pipeline: Python 3.11/3.12/3.13 with uv package manager
    \item Static analysis: mypy (type safety), ruff (linting), bandit (security)
    \item Git LFS integration for binary OTS proofs (prevents repository bloat)
\end{itemize}

\textbf{Documentation completeness:}
\begin{itemize}[nosep]
    \item 9 Architecture Decision Records (ADR-001 through ADR-009)
    \item Operational guides for OTS verification and Bitcoin Core setup
    \item Interactive HTML coverage reports (htmlcov/index.html)
    \item Keep a Changelog format with detailed release notes
\end{itemize}

The M\#4 milestone validates the core thesis: ultra-low-power heritage sensors can achieve cryptographic verifiability without trusted intermediaries. Bitcoin timestamping provides public auditability while preserving payload confidentiality through AEAD encryption.

    \item \textbf{Automated OTS Workflow:} Cron-based anchor and upgrade pipeline
        \begin{itemize}[nosep]
            \item Daily stamping at 00:10 UTC (configurable)

        \end{itemize}

        \end{itemize}
    \item \textbf{Multi-day Verification:} Validate day-chain continuity

            \item Detect chain breaks or root mismatches

        \end{itemize}
\end{itemize}

\textbf{Future Enhancements (M\#6+):}
\begin{itemize}[nosep]
    \item \textbf{Key Rotation:} Implement ADR-001 key derivation schedules
        \begin{itemize}[nosep]
            \item Periodic ratcheting of symmetric keys (e.g., weekly)
            \item Backward secrecy: old keys destroyed after rotation
            \item Forward secrecy: future keys unpredictable from current state
        \end{itemize}
    \item \textbf{Device Attestation:} Ed25519 signature verification
        \begin{itemize}[nosep]
            \item Pod signs frames with device-specific keypair
            \item Gateway validates signatures before accepting frames
            \item Revocation list for compromised device keys
        \end{itemize}
    \item \textbf{Enhanced Proof Metadata:} Richer OTS proof context
        \begin{itemize}[nosep]
            \item Git commit SHA of gateway build
            \item Build timestamp and compiler version
            \item Device firmware versions per pod
        \end{itemize}
    \item \textbf{Air-gapped Deployment Guide:} Offline verification procedures
        \begin{itemize}[nosep]
            \item Pre-downloaded Bitcoin header bundles
            \item Sneakernet workflows for isolated networks
            \item Signature-based header authenticity checks
        \end{itemize}
    \item \textbf{Performance Benchmarking:} Throughput characterization
        \begin{itemize}[nosep]
            \item Frames-per-second ingestion capacity
            \item Merkle batching performance vs. leaf count
            \item OTS verification latency with local vs. remote headers
        \end{itemize}
    \item \textbf{Multi-site Aggregation:} Federated anchoring
        \begin{itemize}[nosep]
            \item Cross-site Merkle trees (site roots as leaves)
            \item Single daily OTS proof for multiple deployment sites
            \item Site isolation while enabling global auditability
        \end{itemize}
\end{itemize}

\subsection{Operational Considerations}

\textbf{Deployment scenarios:}
\begin{itemize}[nosep]
    \item \textbf{Cloud-connected:} Full automation with Bitcoin P2P and public OTS calendars
    \item \textbf{Intermittent connectivity:} Queue OTS operations, batch-process during uplink windows
    \item \textbf{Air-gapped:} Manual OTS export/import via removable media, deferred verification
\end{itemize}

\textbf{Scalability characteristics:}
\begin{itemize}[nosep]
    \item \textbf{Pod count:} Linear growth in frame ingestion (tested up to 10 devices)
    \item \textbf{Frame rate:} Configurable per-pod cadence (typical: 1 frame/5 minutes)
    \item \textbf{Daily batch size:} $O(n \log n)$ Merkle computation (fast for $n < 10^6$ facts)
    \item \textbf{OTS overhead:} One proof per day regardless of fact count (fixed cost)
\end{itemize}

\textbf{Maintenance workflow:}
\begin{itemize}[nosep]
    \item Weekly: Run \texttt{ots upgrade} to update pending proofs with Bitcoin confirmations
    \item Monthly: Backup device table and replay window state
    \item Quarterly: Rotate gateway secrets per key rotation schedule (M\#6)
    \item Annual: Audit complete day-chain for continuity and OTS proof validity
\end{itemize}

\subsection{Research Contributions}

This work presents Track1, a \textbf{system-level integration} of established cryptographic components adapted for a specific deployment niche: heritage sites requiring multi-decade autonomous monitoring without vendor lock-in.

\textbf{Key insight:} Daily Merkle batching amortizes blockchain timestamping costs (1 OTS anchor/day vs. 1000+ individual timestamps) while maintaining per-frame cryptographic integrity. This makes public Bitcoin verification economically viable for resource-constrained IoT deployments.

\textbf{System integration contributions:}

\begin{enumerate}[nosep]
    \item \textbf{Ultra-low power adaptation:} XChaCha20-Poly1305 AEAD with LoRa duty cycle constraints (<1\% EU868), enabling 2+ year battery autonomy on CR2032 cells

    \item \textbf{Deterministic batching:} Canonical JSON + hash-sorted Merkle trees ensure 100\% reproducible roots, enabling independent verification by third parties

    \item \textbf{Zero-marginal-cost anchoring:} Daily OTS batching reduces per-fact blockchain cost from \$0.50 (Ethereum) or \$0.00001 (Helium) to \$0.001/1000 = \$0.000001 per fact

    \item \textbf{Open implementation:} First open-source system combining LoRa + AEAD + Bitcoin OTS with comprehensive test coverage (181 tests, 84\%) and ADR documentation
\end{enumerate}

\textbf{Comparison with prior art:}

\begin{itemize}[nosep]
    \item \textbf{vs. Certificate Transparency / Trillian:} Adapted Merkle timestamping from web PKI to IoT telemetry workflows; Bitcoin verification instead of trusted CT log operators

    \item \textbf{vs. Guardtime KSI:} Open-source implementation with public Bitcoin blockchain instead of proprietary KSI chain and enterprise licensing

    \item \textbf{vs. Helium Network / Ethereum IoT:} 1000× cost reduction through daily batching (\$0.001/day vs. \$0.09-\$1/day); ultra-low power budget (1.4 Wh/day)

    \item \textbf{vs. Commercial IoT platforms:} Vendor-independent verification survives provider discontinuation; no subscription fees or API deprecation risk
\end{itemize}

Track1 targets a \textbf{specific niche} not addressed by existing systems: publicly-auditable telemetry for heritage sites where curators think in decades, not product cycles, and require guarantees that outlive organizational changes.

Track1 achieves its core objective: \emph{verifiable telemetry for resource-constrained sensors without trusted intermediaries}. The M\#4 milestone provides a production-ready foundation with Bitcoin-anchored timestamp proofs and comprehensive test coverage.

The modular architecture and extensive ADR documentation enable independent auditors to validate the system's cryptographic properties and reproduce verification results.

With 158 tests, 85\% coverage, and real Bitcoin attestation confirmed, Track1 is ready for pilot deployments in heritage sensor networks requiring long-term data integrity guarantees.
