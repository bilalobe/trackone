\section{Schemas, Merkle Policy, and OTS (ADR-003)}

\subsection{Canonicalization}
Facts and records are serialized as JSON with:
\begin{itemize}[nosep]
  \item UTF-8, sorted keys, no insignificant whitespace.
  \item Stable number/string encoding; no trailing zeros or locale variance.
\end{itemize}
Leaf hash: $h_i=\mathrm{SHA}\text{-}256(\texttt{canonical\_json}(fact\_i))$.

\subsection{Merkle tree}
\begin{itemize}[nosep]
  \item Sort leaf hashes lexicographically to remove input-order dependence.
  \item Pairwise hash up the tree; when odd, duplicate the last hash at that level.
  \item Root is $\texttt{merkle\_root}$; include leaf\_hashes in the header for auditability.
\end{itemize}

\subsection{Block header schema}
Fields: \texttt{version, site\_id, day, batch\_id, merkle\_root, count, leaf\_hashes}.
Validated by \texttt{block\_header.schema.json}.

\subsection{Day record schema}
Fields: \texttt{version, site\_id, date, prev\_day\_root, batches[], day\_root}.
Canonical day blob written as \texttt{day/YYYY-MM-DD.bin} with a human-readable \texttt{.json} peer.
Validated by \texttt{day\_record.schema.json}.

\subsection{OTS anchoring and verification}
\begin{itemize}[nosep]
  \item Anchor the canonical day blob with OpenTimestamps; store proof alongside the blob.
  \item Verification: recompute Merkle root from canonical facts, check against header/day record, then verify OTS proof.
  \item If OTS client unavailable, write a placeholder and defer confirmation; proofs can be upgraded later.
\end{itemize}

\subsubsection{Local verification workflow}
OpenTimestamps verification requires Bitcoin block headers to validate the Merkle root attestations.
The complete workflow:

\begin{enumerate}[nosep]
  \item \textbf{Generate OTS proof:} \texttt{ots stamp day/2025-10-07.bin} submits hash to calendar servers
  \item \textbf{Upgrade proof:} \texttt{ots upgrade day/2025-10-07.bin.ots} fetches Bitcoin attestations (wait 1--6 hours for block confirmation)
  \item \textbf{Verify proof:} \texttt{ots verify day/2025-10-07.bin.ots} checks Bitcoin block headers contain the committed hash
\end{enumerate}

Example verification output shows multiple attestations across calendar servers and final Bitcoin block confirmations:
\begin{verbatim}
verify BitcoinBlockHeaderAttestation(919388)
verify BitcoinBlockHeaderAttestation(919384)
verify BitcoinBlockHeaderAttestation(919418)
\end{verbatim}

\subsubsection{Bitcoin Core configuration for verification}
Full Bitcoin blockchain sync is impractical for CI environments (831,862 blocks as of October 2025, $>$500 GB). OTS verification only requires \textbf{block headers} (not full blocks).
See ADR-007 for CI policy and operational guidance.

\begin{verbatim}
# Headers-only/pruned mode (< 1 GB storage)
bitcoind -daemon \
  -listen=0 \
  -blocksonly=1 \
  -prune=550 \
  -txindex=0 \
  -dbcache=50 \
  -maxconnections=8
\end{verbatim}

Monitor header sync progress:
\begin{verbatim}
bitcoin-cli -getinfo
# Watch: Verification progress → 100%
# Headers: 919,923 (current tip)
\end{verbatim}

Once headers reach the attested block heights (extracted from \texttt{ots info <file.ots>}), verification succeeds without syncing full blocks.

\subsubsection{CI verification strategy (ADR-007)}
Syncing Bitcoin headers in CI is time-consuming (10--20 minutes for full catchup).
Strategies (per ADR-007):

\begin{itemize}[nosep]
  \item \textbf{Cache Bitcoin datadir:} Persist \texttt{\textasciitilde/.bitcoin} between CI runs to reuse headers
  \item \textbf{Wait for required heights:} Parse \texttt{ots info} output to extract \texttt{BitcoinBlockHeaderAttestation(<height>)}, poll \texttt{bitcoin-cli getblockchaininfo} until \texttt{headers >= max\_height}
  \item \textbf{Timeout handling:} Set reasonable timeout (e.g., 10 minutes) for header sync in CI; on timeout, log ``verification deferred'' and continue
  \item \textbf{Offline verification:} For historical artifacts with confirmed attestations, cache the relevant block headers or skip live verification in CI
\end{itemize}

\textbf{CI script outline (ADR-007):}
\begin{verbatim}
1. Start bitcoind in headers-only mode
2. Extract required block heights from .ots files
3. Wait until headers >= max(required_heights) or timeout
4. Run ots verify on each proof; mark deferred on timeout
5. Cache ~/.bitcoin for next run
\end{verbatim}

\textbf{Trade-off:} OTS provides trustless Bitcoin-anchored timestamps but requires Bitcoin P2P connectivity.
For air-gapped deployments, verification must occur offline with pre-downloaded headers or trusted header feeds.

\subsection{Schema validation}
When \texttt{--validate-schemas} is enabled, validate against:
\begin{itemize}[nosep]
  \item \texttt{fact.schema.json} (telemetry facts)
  \item \texttt{block\_header.schema.json} (batch header)
  \item \texttt{day\_record.schema.json} (daily aggregates)
\end{itemize}
Validation is warning-only in M\#1 to preserve pipeline progress.
