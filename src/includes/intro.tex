\section{Introduction}

Heritage site monitoring presents unique constraints: sensors must operate for years without maintenance, avoid invasive installation, and preserve data integrity across decades without relying on commercial vendors that may discontinue services.
While modern IoT platforms provide telemetry infrastructure, they typically require vendor lock-in (AWS IoT, Azure IoT) or incur per-message costs (blockchain-native systems like Helium Network).

TrackOne addresses a specific niche: \textbf{publicly auditable telemetry for resource-constrained heritage sensors} where multi-decade autonomy is required and vendor lock-in is unacceptable.
By formalizing the telemetry stream as a \textbf{verifiable ledger}, TrackOne ensures that sensor data remains a "source of truth" even if the original gateway operator or cloud infrastructure disappears. This is achieved by anchoring high-frequency facts into daily aggregates, which are then hardened against tampering through a combination of cryptographic Merkle trees and multi-anchor public timestamping.
The system combines ultra-low power LoRa communication (<1kB/day traffic budget), edge AEAD encryption (gateway verifies without cloud dependency), and zero-marginal-cost public timestamping via Bitcoin blockchain through OpenTimestamps.

\subsection{Problem Statement}

Heritage site conservators require telemetry systems that:
\begin{itemize}[nosep]
    \item Operate 2+ years on coin-cell batteries (1 Wh/day budget)
    \item Provide cryptographic data integrity (confidentiality + authenticity)
    \item Enable independent verification without trusting gateway operator
    \item Survive vendor discontinuation (no proprietary cloud dependencies)
    \item Minimize operational costs (zero per-message fees)
\end{itemize}

Existing solutions fail one or more requirements:
\begin{itemize}[nosep]
    \item \textbf{Commercial IoT platforms} (AWS IoT Core, Azure IoT Hub): Vendor lock-in, audit logs controlled by provider, pay-per-message pricing
    \item \textbf{Blockchain-native systems} (Helium Network): Per-packet fees (\$0.00001/packet = \$3.65/year for 1 msg/hour), no power constraints
    \item \textbf{Proprietary timestamping} (Guardtime KSI): Enterprise licensing, closed-source verification, not Bitcoin-anchored
    \item \textbf{Certificate Transparency} (RFC 6962): Designed for TLS certificates, not IoT telemetry workflows
\end{itemize}

\subsection{Contributions}

This work presents TrackOne, an \textbf{open-source implementation} combining low-power cryptography, deterministic Merkle batching, and Bitcoin timestamping for heritage site telemetry.
While individual components are well-established (AEAD encryption, Merkle trees, OpenTimestamps), TrackOne is the \textbf{first open system} integrating them for ultra-low power LoRa deployments with public Bitcoin verification, peer attestation, and optional \textbf{parallel RFC~3161 TSA anchoring} to harden trust against a single gateway operator.

\textbf{Key contributions:}

\begin{enumerate}[nosep]
    \item \textbf{AEAD-protected LoRa frames} (XChaCha20-Poly1305)
    \begin{itemize}[nosep]
        \item Confidentiality + authenticity in <1kB/day traffic budget
        \item Replay protection via a sliding window (64 frames, persistent state)
        \item Compatible with LoRaWAN duty cycle limits (1\% EU868)
        \item Gateway-based verification (no cloud required)
    \end{itemize}

    \item \textbf{Daily Merkle anchoring} via OpenTimestamps
    \begin{itemize}[nosep]
        \item Amortizes blockchain costs: 1 anchor/day vs.
        1000+ individual timestamps
        \item Public verifiability without vendor lock-in (Bitcoin blockchain)
        \item \textbf{Deterministic roots via RFC 8785:} Use of JSON Canonicalization Scheme (JCS) ensures bit-for-bit identical Merkle leaves across Python and Rust implementations, eliminating cross-platform divergence.
        \item Independent verification CLI for third-party auditors
    \end{itemize}

    \item \textbf{Reinforced trust via Parallel RFC 3161 TSA}
    \begin{itemize}[nosep]
        \item Dual-anchoring strategy: Bitcoin (OTS) for decentralization + RFC 3161 for traditional legal/enterprise compliance.
        \item Hardens the "digital contract" against single-point failure of the OTS calendar or Bitcoin P2P network.
        \item Sidecar metadata format allows auditors to choose their preferred trust anchor.
    \end{itemize}

    \item \textbf{Heritage deployment constraints}
    \begin{itemize}[nosep]
        \item 2+ year battery life on CR2032 (220 mAh @ 3 V)
        \item Non-invasive installation (adhesive mount, no wiring)
        \item Open protocols (no proprietary dependencies)
        \item Zero marginal cost per telemetry record
    \end{itemize}

    \item \textbf{Production-ready implementation}
    \begin{itemize}[nosep]
        \item PyNaCl-based gateway with \~85\% test coverage (200+ tests)
        \item Peer attestation toolchain for day roots (Ed25519 signatures, quorum policy)
        \item Optional parallel RFC~3161 TSA anchoring (ADR-015) alongside OTS
        \item CI-verified OTS workflow (Bitcoin Core headers-only mode)
        \item Real Bitcoin attestation: Block 919384 (October 16, 2025)
    \end{itemize}

    \item \textbf{Standardized projections (OGC SensorThings)}
    \begin{itemize}[nosep]
        \item Deterministic mapping of canonical facts to OGC SensorThings API (ADR-028).
        \item Separation of the verifiable "ledger of truth" from the "analytical projection."
        \item Support for daily environmental summaries and stability indices (ADR-029).
    \end{itemize}
\end{enumerate}

\textbf{Novelty claim:} While Certificate Transparency (RFC 6962) and Guardtime KSI demonstrate Merkle timestamping at scale, and blockchain systems like Helium provide decentralized IoT infrastructure, TrackOne is the \textbf{first open-source implementation} specifically scoped for \textbf{multi-decade heritage data sovereignty}. It combines:
\begin{itemize}[nosep]
    \item Ultra-low power radio (LoRa, <1kB/day) with edge AEAD encryption.
    \item RFC 8785-compliant deterministic batching for cross-implementation consistency.
    \item Dual-anchoring (Bitcoin OTS + RFC 3161 TSA) to eliminate vendor and single-protocol lock-in.
\end{itemize}
This targets a \textbf{specific niche}: heritage sites requiring a verifiable ledger of telemetry where data must outlive the hardware and the vendors that produced it.

\subsection{Real-World Impact}

Heritage site conservators think in \textbf{decades}, not product cycles.
TrackOne addresses long-term concerns:

\begin{itemize}[nosep]
    \item \textbf{Vendor discontinuation:} AWS can deprecate IoT services (e.g., AWS IoT 1-Click discontinued 2020).
    Bitcoin blockchain persists 100+ years.
    \item \textbf{Pricing changes:} Cloud providers can increase fees.
    OTS anchoring costs are fixed (Bitcoin transaction fees amortized across all users).
    \item \textbf{Data sovereignty:} Museums require independent verification.
    TrackOne verification CLI runs offline with pre-downloaded Bitcoin headers.
    \item \textbf{Open protocols:} Complete system reproducible from source code.
    No proprietary black boxes.
\end{itemize}
