\section{Introduction}\label{sec:introduction}

Heritage site monitoring presents unique constraints: sensors must operate for years without maintenance, avoid invasive installation, and preserve data integrity across decades without relying on commercial vendors that may discontinue services.
While modern \ac{IoT} platforms provide telemetry infrastructure, they typically require vendor lock-in (AWS \ac{IoT}, Azure \ac{IoT}) or incur per-message costs (blockchain-native systems like Helium Network).

TrackOne addresses a specific niche: \textbf{publicly auditable telemetry for resource-constrained heritage sensors} where multi-decade autonomy is required and vendor lock-in is unacceptable.
By formalizing the telemetry stream as a \textbf{verifiable ledger}, TrackOne ensures that sensor data remains a ``source of truth'' even if the original gateway operator or cloud infrastructure disappears.
This is achieved by anchoring high-frequency facts into daily aggregates, which are then hardened against tampering through a combination of cryptographic Merkle trees and multi-anchor public timestamping.
The system combines ultra-low power \ac{LoRa} communication (<1kB/day traffic budget), edge \ac{AEAD} encryption (gateway verifies without cloud dependency), and zero-marginal-cost public timestamping via Bitcoin blockchain through OpenTimestamps.

% Narrative walkthrough (snappy, end-to-end)
\bigskip
From a single subterranean reading to a publicly verifiable artifact: a pod wakes, captures a measurement, and encrypts it at the edge.
A low-power \ac{LoRa} frame reaches the gateway, which rejects replays and verifies the \ac{AEAD} tag before admitting the fact into the ledger.
The gateway writes a canonical fact, batches the day's facts into a deterministic Merkle tree, and persists a day blob that cannot be silently rewritten.
At day's end, the day root is anchored to public time via OpenTimestamps, producing proofs that survive vendor disappearance.
Weeks later, an auditor reruns \texttt{verify\_cli}: roots recompute, proofs validate, and any tampering is immediately exposed.
This is the core promise: minimal energy, no cloud lock-in, and a durable, public-auditable chain of custody for heritage telemetry.

\subsection{Problem Statement}\label{subsec:problem-statement}

Heritage site conservators require telemetry systems that:
\begin{itemize}[nosep]
    \item Operate on limited power budgets (2+ year battery life)
    \item Provide cryptographic data integrity (confidentiality + authenticity)
    \item Enable independent verification without trusting gateway operator
    \item Survive vendor discontinuation (no proprietary cloud dependencies)
    \item Minimize operational costs (zero per-message fees)
\end{itemize}

Existing solutions fail one or more requirements:
\begin{itemize}[nosep]
    \item \textbf{Commercial IoT platforms} (AWS IoT Core, Azure IoT Hub): Vendor lock-in, audit logs controlled by provider, pay-per-message pricing
    \item \textbf{Blockchain-native systems} (Helium Network): Per-packet fees (\$0.00001/packet = \$3.65/year for 1 msg/hour), no power constraints
    \item \textbf{Proprietary timestamping} (Guardtime KSI): Enterprise licensing, closed-source verification, not Bitcoin-anchored
    \item \textbf{Certificate Transparency} (RFC 6962): Designed for \ac{TLS} certificates, not \ac{IoT} telemetry workflows
\end{itemize}

\subsection{Contributions}\label{subsec:contributions}

This work presents TrackOne, an \textbf{open-source implementation} combining low-power cryptography, deterministic Merkle batching, and Bitcoin timestamping for heritage site telemetry.
While individual components are well-established (\ac{AEAD} encryption, Merkle trees, OpenTimestamps), TrackOne is the \textbf{first open system} integrating them for ultra-low power \ac{LoRa} deployments with public Bitcoin verification, peer attestation, and optional \textbf{parallel RFC~3161 TSA anchoring} to harden trust against a single gateway operator.

\textbf{Key contributions:}

\begin{enumerate}[nosep]
    \item \textbf{AEAD-protected LoRa frames} (XChaCha20-Poly1305)
    \begin{itemize}[nosep]
        \item Confidentiality + authenticity in <1kB/day traffic budget
        \item Replay protection via a sliding window (64 frames, persistent state)
        \item Compatible with \ac{LoRa}WAN duty cycle limits (1\% EU868)
        \item Gateway-based verification (no cloud required)
    \end{itemize}

    \item \textbf{Daily Merkle anchoring} via OpenTimestamps
    \begin{itemize}[nosep]
        \item Amortizes blockchain costs: 1 anchor/day vs.
        1000+ individual timestamps
        \item Public verifiability without vendor lock-in (Bitcoin blockchain)
        \item \textbf{Deterministic roots via TrackOne Canonical CBOR:} canonical commitment bytes ensure bit-for-bit identical Merkle leaves across Python and Rust, eliminating cross-platform divergence.
        \item Independent verification \ac{CLI} for third-party auditors
    \end{itemize}

    \item \textbf{Reinforced trust via Parallel RFC 3161 TSA}
    \begin{itemize}[nosep]
        \item Dual-anchoring strategy: Bitcoin (\ac{OTS}) for decentralization + \ac{RFC} 3161 for legal compliance.
        \item Hardens the digital contract against single-point failure of the \ac{OTS} calendar or Bitcoin network.
        \item Sidecar metadata format allows auditors to choose their preferred trust anchor.
    \end{itemize}

    \item \textbf{Heritage deployment constraints}
    \begin{itemize}[nosep]
        \item 2+ year battery life on a CR2032 bank (multiple cells in parallel; ~2-4 Wh total)
        \item Non-invasive installation (adhesive mount, no wiring)
        \item Open protocols (no proprietary dependencies)
        \item Zero marginal cost per telemetry record
    \end{itemize}

    \item \textbf{Validation-ready implementation}
    \begin{itemize}[nosep]
        \item PyNaCl-based gateway with 85\% test coverage (200+ tests)
        \item Peer attestation toolchain (Ed25519 signatures, quorum policy)
        \item Optional parallel \ac{RFC}~3161 \ac{TSA} anchoring (\ac{ADR}-015) alongside \ac{OTS}
        \item \ac{CI}-verified \ac{OTS} workflow (Bitcoin Core headers-only mode)
        \item Real Bitcoin attestation: Block 919384 (October 16, 2025)
    \end{itemize}

    \item \textbf{Standardized projections (OGC SensorThings)}
    \begin{itemize}[nosep]
        \item Deterministic mapping of facts to \ac{OGC} SensorThings \ac{API} (\ac{ADR}-028).
        \item Separation of "ledger of truth" from "analytical projection."
        \item Support for daily environmental summaries and stability indices (\ac{ADR}-029).
    \end{itemize}
    \end{enumerate}

\subsection{Real-World Impact}\label{subsec:real-world-impact}

Heritage site conservators think in \textbf{decades}, not product cycles.
TrackOne addresses long-term concerns:

\begin{itemize}[nosep]
    \item \textbf{Vendor discontinuation:} AWS can deprecate \ac{IoT} services (e.g., AWS \ac{IoT} 1-Click discontinued 2020).
    Bitcoin blockchain persists 100+ years.
    \item \textbf{Pricing changes:} Cloud providers can increase fees.
    \ac{OTS} costs are fixed (Bitcoin fees amortized).
    \item \textbf{Data sovereignty:} Museums require independent verification.
    Verification \ac{CLI} runs offline with Bitcoin headers.
    \item \textbf{Open protocols:} Complete system reproducible from source code.
    No proprietary black boxes.
    \item \textbf{Energy reality:} A CR2032 bank at roughly 2--4~Wh total is comparable to running a 60~W bulb for about 2--4 minutes (or a 9~W LED for roughly 13--27 minutes), which is the budget TrackOne must stretch across multi-year deployments.
\end{itemize}
