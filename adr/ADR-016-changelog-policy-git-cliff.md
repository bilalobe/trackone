# ADR-016: Changelog Policy and Automation with git-cliff

**Status**: Proposed
**Date**: 2025-11-06
**Milestone Target**: M#5 (post 0.1.0)

## Context

TrackOne currently maintains a manually curated `CHANGELOG.md`. As release cadence accelerates (introducing benchmarking, multi-anchor, Rust core bindings), manual editing becomes error-prone:

- Inconsistent formatting and missing comparison links between versions.
- Risk of omitting security-relevant changes, DB/schema migrations, or deprecations.
- Extra reviewer effort to police commit messages.

We want a reproducible, automated changelog pipeline that:

- Enforces conventional commit semantics (feat, fix, perf, refactor, docs, test, build, ci, chore, security, deprecate).
- Groups entries by type with human-friendly section titles.
- Annotates breaking changes and schema migrations explicitly.
- Provides links to PRs / commits and previous release diff.
- Can be run locally and in CI to gate releases.

## Decision

Adopt `git-cliff` as the canonical tool for generating `CHANGELOG.md` from commit history using a project-specific `cliff.toml` configuration stored in the repo root.

### Scope

- Introduce `cliff.toml` with custom regex for breaking change footers (`BREAKING CHANGE:`) and migration notices (`MIGRATION:`).
- Replace manual editing: developers write conventional commits; release task runs `git cliff --tag <version>` to append a new section.
- CI job validates commit message style and ensures the generated changelog matches the committed file for release PRs.
- Preserve existing historical manual sections (pre- automation) as a static preamble.

### Configuration (Initial `cliff.toml` Sketch)

```toml
[git]
conventional_commits = true
filter_unconventional = true
commit_parsers = [
  { message = "^feat", group = "Features" },
  { message = "^fix", group = "Bug Fixes" },
  { message = "^perf", group = "Performance" },
  { message = "^refactor", group = "Refactoring" },
  { message = "^docs", group = "Documentation" },
  { message = "^test", group = "Tests" },
  { message = "^build", group = "Build" },
  { message = "^ci", group = "CI" },
  { message = "^chore", group = "Chore" },
  { message = "^security", group = "Security" },
  { message = "^deprecate", group = "Deprecations" },
]
protect_breaking_commits = true
filter_commits = true

[git.tag_regex]
pattern = "^v?([0-9]+\.[0-9]+\.[0-9]+)" # supports v1.2.3 or 1.2.3

[changelog]
trim = true
header = "# Changelog\n\nAll notable changes to this project will be documented in this file."
body = "{{#each groups}}{{#if commits.length}}### {{title}}\n{{#each commits}}- {{message}}{{#if breaking}} **(BREAKING)**{{/if}}{{#if footers.MIGRATION}} (Migration: {{footers.MIGRATION}}){{/if}} [{{hash.short}}]({{repo}}/commit/{{hash}}){{/each}}\n\n{{/if}}{{/each}}"
footer = "Generated by git-cliff"
```

### Commit Message Additions

- Breaking change indicator: add `BREAKING CHANGE:` footer describing impact and migration.
- Migration hints: `MIGRATION: <short description>` for schema/state transitions.

## Consequences

### Positive

- Consistent, machine-verifiable changelog sections aligned with semantic versioning.
- Faster releases; less human error.
- Explicit surfacing of breaking changes and migrations for operators.
- Simplified auditing of security fixes and performance improvements.

### Negative / Trade-offs

- Requires discipline in commit messages; learning curve for contributors.
- Rewriting existing non-conforming commits is not retroactive (history stays as-is).
- Additional CI step increases failure surface (style gate).

### Operational Impact

- Add dev dependency (`git-cliff`) or container image in CI.
- Update release checklist (`make release` or GitHub Action) to run generation.
- Documentation update instructing contributors on conventional commits.

## Alternatives Considered

1. Keep manual changelog: rejected (scales poorly, inconsistent).
1. `github-changelog-generator`: GitHub issues/PR driven; less granular grouping and relies on labeling discipline.
1. `towncrier`: fragment-based system; good for large projects but heavier contributor overhead vs conventional commits.
1. Custom script: higher maintenance, duplicates mature tooling.

## Testing & Migration

### Steps

1. Add `cliff.toml` and initial generated changelog for the next release candidate.
1. CI: lint commit messages (regex) and run `git cliff --unreleased` diff; fail if changes found but `CHANGELOG.md` not updated.
1. Tag `v0.1.0` using existing manual log; from `v0.1.1` onward use generated sections.
1. Document conventional commit policy in `CONTRIBUTING.md` referencing this ADR.

### Validation

- Unit: parse `cliff.toml` with `git-cliff --config cliff.toml --unreleased`.
- Integration: simulated release branch; ensure changelog section insertion includes Features/Bug Fixes.
- Manual: verify BREAKING footer surfaces marker and migration hint formatting.

### Rollback

- Keep last good `CHANGELOG.md` and remove `cliff.toml`; drop CI gate. Manual process resumes.

## Future Extensions

- Auto-publish release notes to GitHub Releases using the generated section.
- Add SBOM diff summary group (`sbom:` commits).
- Link issues/PR numbers automatically via regex capture.
- Generate a machine-readable JSON changelog artifact for downstream dashboards.
